# 默认输出目录改造设计

## 目标

在未显式传入 `-out` 时，让解包相关命令默认输出到当前终端工作目录下，而不是输入文件附近。

目标路径格式：

`output/{appid}-{应用名称}/{版本号}/`

约束：

- 只影响 `all` 命令和默认解包命令 `./Gwxapkg -id=... -in=...`
- `repack` 不改
- 用户显式传入 `-out` 时，完全沿用用户给定路径
- 当前工作目录以 `os.Getwd()` 为准
- 版本号无法稳定识别时使用 `unknown`

## 现状

当前默认输出目录由 [internal/cmd/cmd.go](/Users/cai/code/crack/Gwxapkg/internal/cmd/cmd.go) 中的 `DetermineOutputDir` 决定，规则是落在输入目录或输入文件所在目录附近。

这有两个问题：

1. `all` 模式输入的是微信缓存目录，默认输出会直接污染缓存目录附近
2. 默认解包命令输入不固定，输出位置跟输入路径耦合过强，不方便集中管理结果

## 方案

### 路径生成

新增一个专门的默认输出路径 helper，规则如下：

- 根目录：`filepath.Join(cwd, "output")`
- 应用目录：
  - 有名称时：`{appid}-{应用名称}`
  - 无名称时：`{appid}`
- 版本目录：
  - 能识别时：`{版本号}`
  - 否则：`unknown`

名称需要做文件名清洗，至少去掉 `/ \ : * ? " < > |` 等非法字符，并压缩多余空白。

### 元数据来源

`all` 命令：

- 应用名称：直接复用扫描结果里的 `AppName`
- 版本号：直接复用扫描结果里的 `Version`

默认解包命令：

- 应用名称：
  - 优先尝试现有查询缓存/远程查询结果
  - 查不到则留空
- 版本号：
  - `-in` 是目录时，取目录 basename
  - `-in` 是文件时，取父目录 basename
  - basename 是纯数字时视为版本号，否则为 `unknown`

### 调用位置

不去改 `cmd.Execute` 的主流程，而是在 [main.go](/Users/cai/code/crack/Gwxapkg/main.go) 中，当 `-out` 为空时提前算出默认输出目录并传入。

这样做的原因：

- 影响范围最小
- `all` 命令本身已经拿到了应用名称和版本号，不需要把这些上下文硬塞进底层解包逻辑
- 显式 `-out` 和 `repack` 逻辑天然不受影响

## 兼容性与降级

- `os.Getwd()` 失败时，回退到现有默认输出规则，避免命令直接失败
- 应用名称清洗后为空时，目录退化成 `output/{appid}/{版本号}`
- 版本号识别不到时，用 `unknown`
- 现有敏感信息报告、`.gwxapkg` 工作区和恢复后的工程结构都继续落在新的默认输出目录下

## 验证

仅做轻量验证，不新增测试代码：

1. `go run . all -id=<AppID>` 检查默认输出目录是否为 `./output/{appid}-{应用名称}/{版本号}`
2. `go run . -id=<AppID> -in=<版本目录或版本目录下文件>` 检查是否命中对应版本号
3. `go run . -id=<AppID> -in=<非版本路径>` 检查是否落到 `unknown`
